# Подключение Яндекс Облако Object Storage к TrubleBubble

В бэкенде уже используется **S3-совместимый API** (`@aws-sdk/client-s3`). Object Storage в Яндекс Облаке поддерживает те же вызовы — достаточно переменных окружения.

> Условия тарифов, гранта и необходимость привязки карты смотрите на [сайте Яндекс Облака](https://yandex.cloud/docs/) — они могут меняться.

---

## 1. Регистрация и каталог

1. Зайдите в [консоль Яндекс Облака](https://console.cloud.yandex.ru/).
2. Создайте **облако** (cloud), если его ещё нет.
3. Внутри облака создайте **каталог** (folder), например `trublebubble` — запомните **ID каталога** (он понадобится в URL консоли и при привязке прав).

---

## 2. Бакет (bucket)

1. В меню: **Object Storage** → **Бакеты** → **Создать бакет**.
2. Укажите **уникальное имя** (глобально в рамках региона), например `trublebubble-media` — это будет значение **`S3_BUCKET`**.
3. Выберите **зону доступности** (например `ru-central1`) — от неё зависит **`S3_REGION`** (см. ниже).
4. Класс хранилища: обычно **Стандартное** (standard).
5. Создайте бакет.

---

## 3. Публичный доступ к файлам (иначе в браузере будет 403)

Клиент (Netlify) открывает картинки по **прямому HTTPS-URL**. Нужно разрешить **чтение объектов** для всех.

Удобный путь:

1. Откройте бакет → **Права доступа** / **Настройки доступа** (формулировки в консоли могут слегка отличаться).
2. Включите **публичный доступ на чтение** для бакета или настройте **политику доступа (bucket policy)**, разрешающую `s3:GetObject` для всех (`Principal: *`) на объекты в бакете.

Актуальные шаги и пример политики — в документации: [Настройка доступа к бакету](https://yandex.cloud/docs/storage/operations/buckets/edit-acl).

Пока чтение не открыто, загрузка с бэкенда может проходить, но **ссылка в БД не откроется** в браузере.

---

## 4. Статические ключи доступа (Access Key)

Сервер на Render будет ходить в API от имени сервисного аккаунта.

1. **Сервисный аккаунт**: **IAM** (или **Управление доступом**) → **Сервисные аккаунты** → **Создать**.
   - Имя на ваш выбор, например `trublebubble-storage`.
2. Назначьте роль с правами на запись/чтение в Object Storage, например **`storage.editor`** (на каталог или облако — как предлагает мастер).
3. **Где взять ключ в консоли** (названия в интерфейсе могут отличаться):

   - Откройте **IAM** → **Сервисные аккаунты** → **нажмите на имя** созданного аккаунта (не только галочку в списке).
   - На странице аккаунта посмотрите:
     - верхние **вкладки** или **боковое меню** слева;
     - блоки внизу страницы.
   - Ищите пункт в духе: **«Статические ключи доступа»**, **«Ключи доступа»**, **«Access keys»** (английский интерфейс), иногда **«Секреты и ключи»**.
   - Кнопка создания: **«Создать статический ключ»**, **«Создать ключ»**, **«Create access key»**.

   Важно: в шапке консоли должен быть выбран **тот же каталог** (folder), в котором живёт сервисный аккаунт — иначе карточка может открываться «пустой» или без раздела с ключами.

4. Сохраните **идентификатор ключа** и **секрет** — это **`S3_ACCESS_KEY_ID`** и **`S3_SECRET_ACCESS_KEY`**. Секрет показывается **один раз**.

### Если пункта с ключами в консоли нет

- Переключите язык интерфейса (RU/EN) — иногда раздел виден только в одной локали.
- Официальная инструкция (актуальный путь по шагам): [Создать статический ключ доступа](https://yandex.cloud/docs/iam/operations/authentication/manage-access-keys) (раздел *Создать новый ключ*).

### Альтернатива: Yandex Cloud CLI (`yc`)

Удобно, если в веб-интерфейсе раздел спрятан или недоступен.

1. Установите [Yandex Cloud CLI](https://yandex.cloud/docs/cli/quickstart).
2. Выполните `yc init` и авторизуйтесь.
3. Узнайте ID сервисного аккаунта: **IAM** → сервисный аккаунт → поле **ID** в карточке.
4. В терминале:

```bash
yc iam access-key create --service-account-id <ID_СЕРВИСНОГО_АККАУНТА>
```

В ответе будут **`key_id`** (это `S3_ACCESS_KEY_ID`) и **`secret`** (`S3_SECRET_ACCESS_KEY`). Сохраните секрет сразу.

Подробнее: [Создать статический ключ доступа, CLI](https://yandex.cloud/docs/iam/operations/authentication/manage-access-keys).

## 5. Endpoint и регион для AWS SDK

Для **S3 API** в Яндекс Облаке используется:

| Переменная | Типичное значение |
|------------|-------------------|
| **`S3_ENDPOINT`** | `https://storage.yandexcloud.net` |
| **`S3_REGION`** | Код региона бакета, чаще всего **`ru-central1`** (если бакет в этом регионе). Должен совпадать с регионом бакета. |

Список зон и регионов: [документация](https://yandex.cloud/docs/overview/concepts/geo-scope).

В коде проекта для `storage.yandexcloud.net` автоматически включается **`forcePathStyle`** (path-style запросы), как рекомендует совместимость с Яндексом. При необходимости можно переопределить: **`S3_FORCE_PATH_STYLE=true`** или **`false`**.

## 6. Публичный URL для переменной `S3_PUBLIC_BASE_URL`

После загрузки в БД сохраняется полная ссылка вида:

`https://storage.yandexcloud.net/<имя-бакета>/<ключ-объекта>`

Поэтому задайте **базу без завершающего слэша**:

```env
S3_PUBLIC_BASE_URL=https://storage.yandexcloud.net/trublebubble-media
```

Подставьте **своё** имя бакета вместо `trublebubble-media`.

Код добавляет к этой базе ключи вроде `avatars/...` и `chat/<chatId>/...` — итог совпадает с path-style URL Object Storage.

---

## 7. Все переменные для Render (пример)

В **Render** → ваш Web Service → **Environment** добавьте (значения свои):

```env
S3_ENDPOINT=https://storage.yandexcloud.net
S3_REGION=ru-central1
S3_ACCESS_KEY_ID=<идентификатор статического ключа>
S3_SECRET_ACCESS_KEY=<секрет статического ключа>
S3_BUCKET=trublebubble-media
S3_PUBLIC_BASE_URL=https://storage.yandexcloud.net/trublebubble-media
```

Опционально:

```env
S3_KEY_PREFIX=prod
```

Тогда объекты будут с префиксом `prod/avatars/...` — удобно, если в одном бакете несколько окружений.

Локально скопируйте те же ключи в **`backend/.env`** (файл в `.gitignore`).

После деплоя **новые** загрузки пойдут в бакет; старые пути `/uploads/...` в БД останутся на старые файлы (их нужно перезалить или мигрировать отдельно).

---

## 8. Проверка

1. Запустите бэкенд с заполненными переменными.
2. Загрузите аватар в приложении.
3. В консоли Object Storage убедитесь, что появился объект с префиксом `avatars/`.
4. Скопируйте **полный URL** из ответа API или соберите вручную и откройте в браузере — должна открыться картинка **без** авторизации.

Если **403** — снова проверьте публичное чтение бакета/политику.

Если **ошибка подписи или соединения** — проверьте регион, endpoint, ключи и что сервисный аккаунт имеет доступ к этому бакету.

---

## 9. Безопасность

- Не коммитьте **`S3_SECRET_ACCESS_KEY`** в git.
- Статические ключи с правами `storage.editor` — это полный доступ к объектам в рамках прав; при утечке ротируйте ключ в консоли.
