Техническое задание: TrubleBubble
1. Общие положения
1.1 Назначение системы
Данный документ содержит полное техническое задание на разработку кроссплатформенного мессенджера «TrubleBubble» (далее — «Система», «Приложение»). Система представляет собой приложение для мгновенного обмена сообщениями, медиафайлами и документами с функциями создания групп, каналов и конференций.

Интерфейс и пользовательские сценарии ориентированы на официальные клиенты Telegram (UX, терминология, лимиты там, где это указано ниже). Эталонные формулировки и ограничения взяты из публичных материалов Telegram: https://telegram.org/faq (в т. ч. разделы Security, Groups and Channels, Usernames and t.me), https://telegram.org/privacy, https://core.telegram.org/mtproto, https://core.telegram.org/api/end-to-end. Полный перечень URL для сверки — файл `telegram.org` в корне репозитория. TrubleBubble — отдельный продукт; при расхождении с Telegram действует настоящее ТЗ.

1.2 Цели создания системы
Основные цели включают обеспечение надёжной и защищённой коммуникации между пользователями, предоставление интуитивно понятного интерфейса для управления чатами и медиафайлами, а также создание масштабируемой архитектуры, способной обрабатывать миллионы одновременных подключений. Система должна поддерживать синхронизацию данных между устройствами пользователя в режиме реального времени.

1.3 Функциональные требования к системе
Функциональные требования охватывают полный спектр возможностей современного мессенджера: от базовой отправки текстовых сообщений до сложных операций с группами, каналами и медиафайлами. Каждая функция должна быть реализована с учётом требований безопасности, производительности и удобства использования.

1.4 Соответствие модели безопасности и чатов (как у Telegram)
В официальном Telegram облачные чаты (личные и групповые) используют шифрование «клиент–сервер / сервер–клиент» и хранение в облаке; сквозное шифрование (end-to-end) — в секретных чатах и при звонках, с отдельной схемой ключей (MTProto 2.0, см. https://core.telegram.org/api/end-to-end). В TrubleBubble архитектура и API из разделов 2–4 описывают облачную модель сервера; полноценные секретные чаты и криптографическая совместимость с MTProto — вне текущей реализации, см. раздел 6.

1.5 Аутентификация: эталон Telegram и Система
В Telegram основной вход — по мобильному номеру в международном формате и коду подтверждения (SMS или доставка кода через приложение Telegram на связанных устройствах); учётная запись привязана к номеру; опционально — двухэтапная проверка (пароль в дополнение к коду), passkey и др. (см. https://telegram.org/faq#security). Отдельного «входа только по username и паролю» как у классических веб-сервисов в Telegram нет. В Системе сценарии POST /api/auth/send-code и /api/auth/verify моделируют вход по телефону; маршруты login/register по username и паролю — дополнительный сценарий для демонстрации или тестов и не является копией официальной модели входа Telegram.

2. Архитектура системы
2.1 Общая архитектура
Система построена на основе микросервисной архитектуры с разделением на клиентскую и серверную части. Серверная часть обеспечивает хранение данных, бизнес-логику и коммуникацию в реальном времени, тогда как клиентская часть предоставляет пользовательский интерфейс для всех поддерживаемых платформ.

Клиентское приложение реализуется как веб-приложение с использованием современных веб-технологий, адаптированное для работы на настольных компьютерах, планшетах и мобильных устройствах. Серверная часть использует Node.js с Express для REST API и Socket.io для коммуникаций реального времени. База данных PostgreSQL обеспечивает надёжное хранение структурированных данных, а Redis используется для кэширования сессий и Pub/Sub коммуникаций.

2.2 Структура базы данных
Таблица пользователей (users)
Таблица users содержит полную информацию о каждом пользователе системы. Поля включают уникальный идентификатор (uuid), номер телефона в международном формате со значением unique и опциональным флагом verified, имя пользователя (username) в нижнем регистре со значением unique для входа по ссылке, отображаемое имя (display_name) для показа в интерфейсе, URL аватара профиля (avatar_url), биографию пользователя (bio) с максимальной длиной 70 символов, хеш пароля (password_hash) для аутентификации по логину и паролю, флаг онлайн-статуса (is_online) булевого типа, временную метку последнего входа (last_seen_at), а также стандартные временные метки created_at и updated_at для отслеживания изменений.

Таблица чатов (chats)
Таблица chats хранит информацию о всех чатах системы. Поля включают уникальный идентификатор (uuid), тип чата (type) из перечисления PRIVATE, GROUP, CHANNEL, SUPERGROUP, название чата или группы (title) с максимальной длиной 128 символов, URL аватара чата (avatar_url), описание чата или канала (description) с максимальной длиной 255 символов, уникальное имя пользователя (username) для публичных чатов и каналов, ссылку на создателя чата (created_by_id) как внешний ключ к таблице users, а также стандартные временные метки. Эталон Telegram: группы — до 200 000 участников; каналы — неограниченное число подписчиков (https://telegram.org/faq#groups-and-channels). Лимиты Системы должны быть явно заданы в конфигурации и не обязаны достигать этих верхних границ на первом этапе.

Таблица участников чатов (chat_members)
Таблица chat_members связывает пользователей с чатами и определяет их роли. Поля включают внешний ключ chat_id со ссылкой на chats с каскадным удалением, внешний ключ user_id со ссылкой на users с каскадным удалением, роль участника (role) из перечисления OWNER, ADMIN, MEMBER со значением по умолчанию MEMBER, временную метку входа в чат (joined_at) со значением по умолчанию текущего времени. Комбинация chat_id и user_id является первичным ключом.

Таблица сообщений (messages)
Таблица messages хранит все сообщения системы. Поля включают уникальный идентификатор (uuid), внешний ключ chat_id со ссылкой на chats с каскадным удалением, внешний ключ sender_id со ссылкой на users, текстовое содержание сообщения (content) с типом text, тип содержимого (content_type) из перечисления TEXT, IMAGE, VIDEO, FILE, AUDIO, VOICE со значением по умолчанию TEXT, URL медиафайла (media_url), размер медиафайла в байтах (media_size), ссылку на родительское сообщение для ответов (reply_to_id), флаг редактирования (is_edited) булевого типа со значением по умолчанию false, флаг удаления (is_deleted) булевого типа со значением по умолчанию false, а также стандартные временные метки. Создаётся индекс по паре (chat_id, created_at) для эффективной загрузки истории.

Таблица статусов сообщений (message_statuses)
Таблица message_statuses отслеживает статус доставки и прочтения сообщений для каждого пользователя. Поля включают внешний ключ message_id со ссылкой на messages с каскадным удалением, внешний ключ user_id со ссылкой на users, статус (status) из перечисления SENT, DELIVERED, READ со значением по умолчанию SENT, временную метку обновления статуса (updated_at). Комбинация message_id и user_id является первичным ключом.

Таблица ключей шифрования (encryption_keys)
Таблица encryption_keys зарезервирована под будущую реализацию защиты на клиенте (например, обмен ключами для режима, аналогичного секретным чатам Telegram по смыслу). Поля включают внешний ключ user_id как первичный ключ со ссылкой на users с каскадным удалением, публичный ключ (public_key) текстового типа, зашифрованный приватный ключ (private_key_encrypted), временную метку обновления (updated_at). В Telegram сквозное шифрование сообщений реализовано в рамках MTProto Secret Chats (https://core.telegram.org/api/end-to-end), а не путём хранения пары ключей только в таблице на сервере.

2.3 API эндпойнты
Аутентификация (Auth)
Маршрут POST /api/auth/send-code принимает номер телефона и отправляет проверочный код. Валидация включает проверку формата номера телефона, ограничение частоты запросов до 3 попыток на номер за 10 минут, возврат одноразового 6-значного кода. Ответ включает флаг success, сообщение об успешной отправке и демонстрационный код в dev-режиме.

Маршрут POST /api/auth/verify принимает номер телефона, проверочный код, имя и фамилию (опционально), username (опционально). Валидация проверяет совпадение кода, создаёт нового пользователя при первом входе или возвращает существующего. Ответ включает объект user, accessToken со сроком действия 15 минут, refreshToken со сроком действия 7 дней.

Маршрут POST /api/auth/login принимает username и password. Валидация проверяет наличие пользователя и совпадение пароля с использованием bcrypt. Ответ идентичен verify.

Маршрут POST /api/auth/register принимает username, password, displayName. Валидация проверяет уникальность username (только a-z, 0-9, подчёркивание), минимальную длину пароля 6 символов. Ответ идентичен verify.

Маршрут POST /api/auth/refresh принимает refreshToken. Валидация проверяет валидность токена и существование пользователя. Ответ включает новые accessToken и refreshToken.

Маршрут POST /api/auth/logout инвалидирует refreshToken на сервере и возвращает флаг успеха.

Пользователи (Users)
Маршрут GET /api/users/me возвращает данные текущего пользователя на основе JWT токена. Ответ включает все поля пользователя кроме passwordHash.

Маршрут PATCH /api/users/me принимает displayName, bio, avatarUrl. Обновляет профиль текущего пользователя и возвращает обновлённые данные.

Маршрут GET /api/users/search принимает параметр q для поискового запроса. Возвращает список пользователей, исключая текущего, где username, displayName или phone содержат запрос. Ограничено 20 результатами.

Маршрут GET /api/users/:id возвращает данные пользователя по идентификатору. Если пользователь не найден, возвращается ошибка 404.

Чаты (Chats)
Маршрут GET /api/chats возвращает список чатов текущего пользователя с информацией о последнем сообщении и количеством непрочитанных. Сортировка по updatedAt в порядке убывания.

Маршрут POST /api/chats принимает type, title (для GROUP, CHANNEL), memberIds (для PRIVATE), username (опционально). Для PRIVATE чатов проверяется существование чата между участниками и возвращается существующий или создаётся новый. Для GROUP создаётся группа с текущим пользователем как OWNER. Ответ включает полную информацию о чате с участниками.

Маршрут GET /api/chats/:id возвращает информацию о чате с участниками. Проверяется членство текущего пользователя в чате.

Маршрут PATCH /api/chats/:id принимает title, description, avatarUrl. Обновляет информацию о чате. Требуются права ADMIN или OWNER.

Маршрут DELETE /api/chats/:id удаляет чат. Только OWNER может удалить чат.

Маршрут POST /api/chats/:id/members принимает массив userIds. Добавляет участников в чат. Требуются права ADMIN или OWNER.

Маршрут DELETE /api/chats/:id/members/:userId удаляет участника из чата. Участник может удалить себя, ADMIN или OWNER могут удалить любого кроме OWNER.

Маршрут POST /api/chats/:id/leave позволяет текущему пользователю покинуть чат. OWNER не может покинуть чат без передачи прав.

Сообщения (Messages)
Маршрут GET /api/messages/chat/:chatId принимает необязательные параметры before (ISO дата для пагинации) и limit (по умолчанию 50, максимум 100). Возвращает сообщения чата в обратном хронологическом порядке. Помечает сообщения как прочитанные для текущего пользователя.

Маршрут POST /api/messages/chat/:chatId принимает content, contentType (по умолчанию TEXT), mediaUrl, mediaSize, replyToId (опционально). Создаёт сообщение, инициализирует статусы SENT для всех участников кроме отправителя. Возвращает созданное сообщение с информацией об отправителе.

Маршрут PATCH /api/messages/:id принимает content. Редактирует сообщение в течение 48 часов после создания. Только отправитель может редактировать. Устанавливает флаг isEdited.

Маршрут DELETE /api/messages/:id принимает необязательный параметр deleteForEveryone. При значении true удаляет сообщение для всех, иначе только для текущего пользователя. Проверяются права (отправитель или ADMIN).

Маршрут POST /api/messages/chat/:chatId/read принимает необязательный messageId. Помечает все сообщения или конкретное сообщение как прочитанные.

Маршрут GET /api/messages/search принимает параметры q (поисковый запрос) и необязательный chatId. Возвращает сообщения содержащие запрос. Ограничено 50 результатами.

2.4 WebSocket события
Клиент → Сервер
Событие join_chat с параметром chatId присоединяет клиента к комнате чата. Сервер проверяет членство и добавляет клиента в комнату Socket.io.

Событие leave_chat с параметром chatId покидает комнату чата.

Событие send_message с параметрами chatId, content, contentType, mediaUrl, mediaSize, replyToId создаёт сообщение и широковещательно рассылает его участникам чата через new_message.

Событие typing_start с параметром chatId уведомляет участников о начале печати.

Событие typing_stop с параметром chatId уведомляет участников о завершении печати.

Событие mark_read с параметрами chatId и необязательным messageId помечает сообщения как прочитанные и уведомляет отправителя.

Событие edit_message с параметрами messageId и content редактирует сообщение и рассылает обновление.

Событие delete_message с параметрами messageId и deleteForEveryone удаляет сообщение и рассылает уведомление.

Сервер → Клиент
Событие new_message с данными сообщения рассылается всем участникам чата.

Событие message_edited с обновлёнными данными сообщения рассылается участникам чата.

Событие message_deleted с messageId и chatId рассылается участникам чата.

Событие message_sent с messageId и chatId отправляется отправителю для подтверждения.

Событие message_delivered с messageId и chatId отправляется отправителю при доставке получателю.

Событие message_read с messageId, userId и chatId отправляется отправителю сообщения.

Событие typing с chatId, userId и isTyping отправляется участникам чата.

Событие user_online с userId рассылается всем при подключении пользователя.

Событие user_offline с userId рассылается всем при отключении пользователя.

3. Пользовательский интерфейс
3.1 Экран входа и регистрации
3.1.1 Авторизация по номеру телефона
Экран авторизации по номеру телефона состоит из следующих элементов. Логотип приложения «TrubleBubble» отображается в центре верхней части экрана с акцентным синим цветом в духе Telegram (#0088CC) и название продукта под ним. Поле ввода номера телефона принимает международный формат (+1234567890), как рекомендует Telegram для регистрации (см. https://telegram.org/faq#login-and-sms — мобильный номер, не городской), с автоматическим определением страны по коду. Placeholder показывает пример номера. При вводе автоматически добавляются пробелы и скобки для удобства чтения.

Кнопка «Далее» становится активной при вводе корректного номера (минимум 10 цифр). При нажатии отправляется запрос на отправку кода и выполняется переход к экрану ввода кода. Кнопка имеет фирменный синий цвет, скруглённые углы (8px) и состояние disabled при невалидном номере.

Ссылка «Войти по имени пользователя» переключает на экран входа по username и паролю. Размещена под основной кнопкой, имеет контрастный цвет и подчёркивание при наведении.

3.1.2 Экран ввода проверочного кода
После отправки кода отображается экран подтверждения. Заголовок показывает «Код подтверждения» с подзаголовком о отправке SMS на указанный номер. При клике на номер открывается возможность изменить его.

Поле ввода кода принимает 6 цифр с автоматическим переходом к следующему символу. Используется моноширинный шрифт увеличенного размера для удобства чтения. Каждая введённая цифра отображается немедленно.

При успешной проверке кода (любой 6-значной комбинации в демо-режиме) отображается экран создания профиля для новых пользователей. Для существующих пользователей выполняется переход к списку чатов.

Кнопка «Назад» позволяет вернуться к вводу номера телефона.

3.1.3 Экран создания профиля
Для новых пользователей отображается форма создания профиля. Поле «Ваше имя» обязательно для заполнения, максимум 64 символа. Используется для отображения в чатах и сообщениях.

Поле «Username» опционально, должно быть уникальным; допустимые символы как в Telegram: латиница a–z, цифры 0–9 и подчёркивание; регистр не различается при поиске, отображение регистра может сохраняться (https://telegram.org/faq#q-what-can-i-use-as-my-username). Минимальная длина 5 символов — требование Системы. Проверка уникальности выполняется в реальном времени с индикатором доступности.

Кнопка «Начать общение» создаёт аккаунт и переходит к списку чатов.

3.1.4 Авторизация по username
Экран входа по username включает поле username с автозаполнением и историей входов. Поле password с маской символов и кнопкой показа/скрытия. Кнопка «Войти» активна только при заполненных полях. Ссылка «Войти по номеру» возвращает к авторизации по телефону.

3.2 Основной экран — список чатов
3.2.1 Шапка приложения
Шапка содержит аватар текущего пользователя (40x40px, круглый) в левой части. При клике открывается меню профиля или переход к настройкам. Индикатор онлайн-статуса (зелёная точка 10px) в правом нижнем углу аватара.

Справа располагается кнопка меню (иконка «три точки») для доступа к настройкам и другим функциям.

3.2.2 Строка поиска
Поле поиска расположено под шапкой с иконкой лупы слева. Placeholder «Поиск или начать новый чат». При фокусе поле подсвечивается рамкой primary цвета. Поиск работает по именам контактов, usernames и названиям групп.

Результаты поиска отображаются в выпадающем списке под полем с группировкой по категориям: контакты, группы, каналы. При клику на результат открывается чат или начинается новый диалог.

3.2.3 Список чатов
Список чатов занимает основное пространство экрана. Каждый элемент списка представляет один чат и включает несколько компонентов.

Аватар чата или контакта размером 48x48px отображается слева. Для личных чатов показывается фото контакта или заглушка с первой буквой имени. Для групп отображается фото группы или иконка группы. Для каналов отображается логотип канала или иконка канала.

Индикатор онлайн-статуса (зелёная точка 12px) позиционируется в правом нижнем углу аватара для личных чатов с онлайн контактами.

Основная информация о чате располагается справа от аватара и включает имя контакта или название группы (жирный шрифт для непрочитанных), последнее сообщение или «Ничего ещё» для новых чатов, временную метку последнего сообщения (справа вверху, серый цвет).

Для непрочитанных чатов отображается счётчик непрочитанных сообщений (красный круг с белым числом, максимум «99+») справа внизу блока информации. Сообщения отображаются серым цветом и обрезаются одной строкой с многоточием.

При длительном нажатии или клике правой кнопкой мыши открывается контекстное меню с опциями: закрепление чата, отключение уведомлений, удаление чата, блокировка пользователя (для личных чатов).

3.2.4 Пустое состояние
При отсутствии чатов отображается пустое состояние с иконкой сообщений (серый цвет, 64px). Текст «Нет чатов» заголовком и подзаголовок «Начните общение, нажав кнопку ниже». CTA кнопка «Новое сообщение» переводит к созданию диалога.

3.2.5 Кнопка создания чата
Плавающая кнопка или кнопка в нижней части экрана для создания нового чата. Иконка карандаша или плюса на фирменном синем фоне. При клике открывается модальное окно поиска пользователей для создания диалога.

3.3 Экран чата
3.3.1 Шапка чата
Шапка чата фиксирована в верхней части и содержит следующие элементы. Кнопка возврата (стрелка назад) слева, отображается на мобильных устройствах и скрывается на десктопе при видимой боковой панели.

Аватар контакта или группы (40x40px) кликабелен для просмотра профиля. Для личных чатов рядом с аватаром отображается имя контакта (жирный) и статус (онлайн/был(а) недавно/оффлайн). Для групп отображается название группы и количество участников.

Группа кнопок действий справа включает поиск по чату (иконка лупы), звонок (иконка телефона, будущая функция), видеозвонок (иконка видео, будущая функция), меню чата (иконка «три точки»).

При клике на аватар или имя открывается профиль контакта или информация о группе.

3.3.2 Область сообщений
Область сообщений занимает всё доступное пространство между шапкой и полем ввода. Сообщения группируются по дате с разделителями.

Разделитель даты отображается как горизонтальная линия с текстом даты в центре (белый текст на полупрозрачном фоне). Для сегодняшней даты показывается «Сегодня», для вчера — «Вчера», для остальных — формат «21 марта 2024».

Каждое сообщение располагается в пузыре с следующими характеристиками. Исходящие сообщения (отправленные текущим пользователем) выравниваются справа и имеют синюю заливку (#0088CC). Входящие сообщения выравниваются слева и имеют тёмную заливку (#182533).

Текст сообщения отображается белым цветом, поддерживает перенос строк и emoji. Максимальная длина 4096 символов для текста.

Время отправки показывается в правом нижнем углу пузыря (серый текст, 11px). Для исходящих сообщений рядом отображается статус доставки и прочтения в терминах Telegram (https://telegram.org/faq#q-what-do-the-check-marks-mean): одна галочка — сообщение доставлено в облако, собеседник уведомлён (если у него включены уведомления); две галочки — сообщение прочитано (собеседник открыл чат и сообщение). Отдельного статуса «доставлено на конкретное устройство» в Telegram нет из‑за синхронизации на нескольких устройствах — в интерфейсе Системы также не дублировать трёхстадийную схему «отправлено / доставлено / прочитано» в стиле других мессенджеров, если это противоречит двум галочкам.

Индикатор редактирования «(ред.)» отображается после времени для отредактированных сообщений.

Для ответов на сообщения сверху пузыря отображается цитата с вертикальной линией слева ( accent цвет), именем автора оригинального сообщения и текстом (обрезается до одной строки).

Медиафайлы отображаются инлайн: изображения с предпросмотром (максимальная ширина 260px, сохраняется пропорция), видео с превью и длительностью, файлы с иконкой типа и названием.

3.3.3 Индикатор печати
При получении события typing от другого пользователя в области сообщений отображается «печатает...» с анимированными точками. Текст серого цвета, располагается над полем ввода.

3.3.4 Поле ввода сообщения
Поле ввода сообщения фиксировано в нижней части чата. Фон — #1F2C33, разделитель сверху — тонкая линия.

Слева располагается кнопка прикрепления файлов (иконка скрепки). При клике открывается меню с опциями: фото или видео (открывает галерею), камера (открывает камеру устройства), файл (открывает проводник), геолокация (заглушка).

Поле ввода текста автоматически расширяется по мере ввода (максимум 5 строк). Поддерживает многострочный ввод. Placeholder «Сообщение» серым цветом.

При ответе на сообщение над полем ввода отображается превью оригинального сообщения с кнопкой отмены (иконка X).

Кнопка отправки располагается справа. При наличии текста отображается иконка самолёта, при отсутствии — иконка микрофона. Кнопка активируется при наличии текста или медиафайла.

Кнопка отправки становится активной (фирменный синий цвет) только при наличии текста или прикреплённого файла. Показ превью прикреплённого изображения или файла слева от поля ввода.

3.4 Экран профиля пользователя
3.4.1 Шапка профиля
Шапка с кнопкой возврата (←) и заголовок «Профиль». Минимальное разворачивание на десктопе.

3.4.2 Блок аватара
Большой аватар пользователя (200x200px на десктопе, на всю ширину на мобильных) с кнопкой смены фото (иконка камеры). При клике на аватар открывается полноразмерное изображение.

Для собственного профиля кнопка «Загрузить фото» или «Удалить фото».

3.4.3 Информация о пользователе
Блок с именем пользователя (крупный шрифт, 24px), username (@username, серый цвет), статус (онлайн/был недавно).

Для личных чатов отображается биография пользователя, если указана.

3.4.4 Блок контактной информации
Секция с контактной информацией включает номер телефона (если пользователь разрешил показ), username (если есть), e-mail (если добавлен), дата регистрации («В Telegram с марта 2024»).

Каждый элемент кликабелен: номер телефона — копирование или звонок (заглушка), username — копирование, дата — копирование. Формулировка даты регистрации в духе «В приложении с …» (без товарного знака Telegram).

3.4.5 Медиа, файлы, ссылки
Вкладки для переключения между разделами: медиафайлы, файлы, ссылки. По умолчанию выбраны медиафайлы.

Сетка превью медиафайлов (3 колонки на десктопе, 2 на мобильных). При клику на элемент открывается полноразмерный просмотр.

3.4.6 Кнопки действий
Блок с кнопками действий для личных чатов включает «Сообщение» (открывает чат), «Позвонить» (заглушка), «Видеозвонок» (заглушка).

Кнопка «Заблокировать пользователя» красного цвета для личных чатов. При блокировке пользователь исчезает из списка контактов и не может писать.

Кнопка «Пожаловаться» для сообщения о нарушениях.

3.5 Экран создания группы
3.5.1 Шаг 1: Название и фото
Форма создания группы включает аватар-заглушку с иконкой группы, возможность загрузки фото, поле названия группы (обязательно, макс 128 символов), поле username (опционально, уникальность проверяется).

3.5.2 Шаг 2: Добавление участников
Поиск и выбор участников из списка контактов. Галочки напротив выбранных. Список выбранных участников с возможностью удаления.

Счётчик выбранных участников.

3.5.3 Кнопка создания
Кнопка «Создать группу» активна при наличии названия и минимум одного участника. Создаёт чат с типом GROUP и перенаправляет в него.

3.6 Экран настроек
3.6.1 Структура настроек
Настройки организованы в секции с заголовками: аккаунт, чаты, уведомления и звуки, приватность и безопасность, расширенные, внешний вид, язык, справочник.

Каждый пункт меню отображается как строка с иконкой слева, названием и Chevron справа. При клике открывается соответствующий экран настроек.

3.6.2 Настройки аккаунта
Пункт «Изменить профиль» открывает форму редактирования профиля. Пункт «Номер телефона» показывает текущий номер и позволяет изменить (требует верификации). Пункт «Имя пользователя» позволяет изменить username. Пункт «Удалить аккаунт» требует подтверждения с вводом пароля.

3.6.3 Настройки чатов
Пункт «Размер шрифта в чатах» открывает выбор размера (маленький, средний, большой, очень большой). Пункт «Стикер-паки» открывает менеджер стикеров. Пункт «Создать группу» открывает мастер создания группы. Пункт «Создать канал» открывает мастер создания канала.

3.6.4 Настройки уведомлений и звуков
Пункт «Уведомления» переключатель вкл/выкл для всех чатов. При выключении отображается «Выкл.», при включении — «Вкл.».

Пункт «Звук уведомлений» выбор звука из предустановленного списка или «Без звука». Preview воспроизводит выбранный звук.

Пункт «Вибрация» переключатель вкл/выкл.

Пункт «Превью сообщения» показ текста сообщения в уведомлении или только отправитель. Опции: «Всегда», «Контакты», «Никогда».

Пункт «Настройки для отдельных чатов» открывает список чатов с возможностью индивидуальной настройки.

3.6.5 Приватность и безопасность
Пункт «Двухэтапная проверка» (аналог Two-Step Verification в Telegram: код по SMS плюс пароль, см. https://telegram.org/faq#q-how-does-2-step-verification-work) — защита аккаунта паролем. При включении запрашивается пароль и подсказка.

Пункт «Код доступа» блокировка приложения PIN-кодом или Face ID / Touch ID.

Пункт «Активные сессии» список устройств с активными сессиями. Показывает устройство, местоположение (IP), время последней активности. Кнопка «Завершить» для каждой сессии, «Завершить все остальные» для всех кроме текущей.

Пункт «Кто видит мой номер» настройка видимости: все, контакты, никто.

Пункт «Кто видит мои последние посещения» аналогично для статуса онлайн.

Пункт «Пересылка сообщений» опция «Кто может ссылаться на мои сообщения» везде: все, контакты, никто.

Пункт «Заблокированные пользователи» список заблокированных с возможностью разблокировки.

3.6.6 Расширенные настройки
Пункт «Автозагрузка медиа» выбор условий автозагрузки: только по Wi-Fi, всегда, вручную. Настройка для каждого типа: фото, видео, файлы.

Пункт «Использовать меньше трафика» экономия трафика при передаче медиа.

Пункт «Папки чатов» (future) — организация чатов по папкам (аналог папок в клиентах Telegram).

Пункт «Хранилище и данные» управление кэшем, историй, загрузками.

3.6.7 Внешний вид
Пункт «Тема» выбор цветовой схемы: светлая, тёмная, системная. На десктопе доступна настройка размера чата.

Пункт «Обои» выбор обоев для чата из предустановленных или своих.

Пункт «Размер чата» слайдер для настройки размера текста в чате.

3.6.8 Язык
Пункт «Язык приложения» выбор языка интерфейса из списка поддерживаемых.

4. Функциональные требования
4.1 Управление пользователями
4.1.1 Регистрация нового пользователя
Регистрация начинается с ввода номера телефона. Система валидирует формат номера, генерирует 6-значный код и отправляет его (в демо-режиме код выводится в консоль). Пользователь вводит код, при успешной проверке создаётся запись в базе данных с номером телефона и временной меткой. Новый пользователь получает статус онлайн и может настроить профиль.

Альтернативный путь — регистрация по username и паролю. Пользователь выбирает уникальный username (5-32 символа, только a-z, 0-9, _) и пароль (минимум 6 символов). Пароль хешируется с использованием bcrypt и сохраняется в базе. После регистрации автоматически выполняется вход.

4.1.2 Авторизация пользователя
При входе по номеру телефона выполняется проверка кода и поиск существующего пользователя. При первом входе создаётся новый пользователь. При входе по username/password выполняется проверка учётных данных. После успешной аутентификации генерируются JWT токены: accessToken (15 минут) и refreshToken (7 дней). Токены сохраняются в localStorage/sessionStorage на клиенте.

4.1.3 Редактирование профиля
Пользователь может изменить отображаемое имя (1-64 символа), username (проверка уникальности), биографию (максимум 70 символов), аватар (загрузка изображения с валидацией формата и размера). Изменения сохраняются через PATCH /api/users/me и синхронизируются на всех устройствах.

4.1.4 Удаление аккаунта
Удаление аккаунта требует подтверждения с вводом пароля. При удалении выполняются следующие операции: удаление данных пользователя в соответствии с политикой Системы, удаление из всех чатов, удаление ключей шифрования, удаление токенов авторизации, удаление записи пользователя. В Telegram при удалении аккаунта сообщения в группах остаются у других участников; собеседники сохраняют копии переписки (https://telegram.org/faq#q-what-happens-if-i-delete-my-account). В Системе для групповых чатов допускается аналогичная модель: сообщения в общих чатах не стираются у остальных без отдельного сценария; у собеседников может отображаться нейтральная подпись вместо имени удалённого пользователя.

4.2 Управление чатами
4.2.1 Создание личного чата
Личный чат создаётся при выборе пользователя из списка контактов или поиска. Система проверяет существование чата между участниками. Если чат существует, возвращается ссылка на него. Если нет, создаётся новый чат с типом PRIVATE и участниками — создатель и выбранный пользователь. Чат появляется в списке обоих пользователей.

4.2.2 Создание группы
Создание группы начинается с указания названия (обязательно) и загрузки аватара (опционально). Затем выбираются участники из списка контактов (минимум 1). При создании устанавливается тип SUPERGROUP (в Telegram крупные группы с единой историей и до 200 000 участников; см. https://telegram.org/faq#q-what-makes-telegram-groups-cool). Создатель получает роль OWNER. Публичная ссылка на чат в эталоне Telegram — t.me/username или t.me/slug для группы/канала; в Системе — уникальная ссылка-приглашение, совместимая по смыслу с пригласительными ссылками (https://telegram.org/blog/invite-links).

4.2.3 Создание канала
Создание канала аналогично группе, но с типом CHANNEL. Создатель становится единственным автором (может публиковать). Остальные пользователи — подписчики (могут только читать). В Telegram у канала неограниченное число подписчиков; сообщения подписаны именем канала; у сообщений есть счётчик просмотров (https://telegram.org/faq#q-what-39s-the-difference-between-groups-and-channels). Канал может быть публичным (с username для поиска) или приватным (только по приглашению).

4.2.4 Управление участниками
Добавление участников доступно для ADMIN и OWNER. Выполняется поиск пользователей и добавление выбранных. Новые участники видят всю историю сообщений (если не ограничено настройками).

Удаление участников: ADMIN и OWNER могут удалять MEMBER, OWNER может удалять ADMIN. Участник может удалить себя (покинуть чат). OWNER не может покинуть чат — должен передать права или удалить чат.

Назначение администраторов доступно только OWNER. ADMIN получает расширенные права: добавление/удаление участников, редактирование информации о чате, удаление сообщений.

4.2.5 Редактирование чата
Редактирование доступно для ADMIN и OWNER. Можно изменить название (1-128 символов), описание (до 255 символов), аватар. Изменения применяются немедленно и отображаются у всех участников.

4.3 Управление сообщениями
4.3.1 Отправка текстового сообщения
Пользователь вводит текст в поле ввода и нажимает Enter или кнопку отправки. Текст валидируется (не пустой, максимум 4096 символов). Сообщение сохраняется в базе данных со статусом SENT для всех участников. Socket.io уведомляет участников чата о новом сообщении. Отправитель получает подтверждение доставки.

4.3.2 Отправка медиафайлов
Прикрепление файлов выполняется через меню attachment. В Telegram один пересылаемый файл любого типа может весить до 2 GB (https://telegram.org/faq#q-what-makes-telegram-groups-cool, сравнение с WhatsApp на https://telegram.org/faq#q-how-is-telegram-different-from-whatsapp). Поддерживаемые типы в Системе: изображения (JPEG, PNG, GIF, WebP до 10MB для превью в лимитах сервера), видео (MP4 до 2GB), аудио (MP3, OGG до 2GB), прочие файлы (до 2GB), если инфраструктура хранения позволяет.

Загрузка файла выполняется следующим образом: выбирается файл, показывается прогресс загрузки, файл загружается на сервер или S3-совместимое хранилище, создаётся сообщение с типом контента и URL файла, файл становится доступен получателям.

Превью изображения отображается инлайн в чате. Для видео показывается превью с длительностью. Для файлов отображается иконка типа и название.

4.3.3 Ответы на сообщения
При клике на сообщение и выборе «Ответить» поле ввода дополняется блоком reply с информацией об оригинальном сообщении. Отправленное сообщение содержит ссылку replyToId на оригинальное. В чате отображается цитата оригинального сообщения над текстом ответа.

4.3.4 Редактирование сообщений
В Telegram сообщения в облачных чатах можно редактировать после отправки (в т. ч. в группах; см. https://telegram.org/faq#q-what-makes-telegram-groups-cool). В Системе редактирование доступно только автору в течение 48 часов после отправки — ограничение реализации, а не дословная копия политики клиента Telegram. При редактировании текст обновляется, устанавливается флаг isEdited, всем участникам рассылается событие message_edited. В интерфейсе рядом с временем отображается «(ред.)».

4.3.5 Удаление сообщений
В личном чате Telegram позволяет удалять отправленные и полученные сообщения для обеих сторон без следа в чате (https://telegram.org/faq#q-can-i-delete-my-messages). В группах — удаление своих сообщений. В Системе: удаление для себя скрывает сообщение только у удаляющего пользователя (устанавливается флаг isDeleted, текст заменяется на «Сообщение удалено»). Удаление для всех доступно автору и администраторам там, где предусмотрено правилами чата. При удалении для всех сообщение удаляется у всех участников.

4.3.6 Статусы сообщений
Статус SENT устанавливается при создании сообщения. Статус DELIVERED устанавливается при получении события получения сервером (Socket.io) — соответствует смыслу «доставлено в облако» для одной галочки в Telegram. Статус READ устанавливается при открытии чата получателем и отправке события mark_read — соответствует двум галочкам в Telegram. Статусы визуализируются галочками рядом с исходящими сообщениями согласно п. 3.3.2.

4.4 Поиск
4.4.1 Глобальный поиск
Поиск выполняется по всем чатам сразу. Ищется текст в сообщениях, имена контактов, usernames, названия групп и каналов. Результаты группируются по типу: контакты, группы, сообщения. Показывается контекст вокруг найденного текста. Клик на результат открывает чат и прокручивает к сообщению.

4.4.2 Поиск в чате
Поиск внутри конкретного чата аналогичен глобальному, но ограничен историей этого чата. Дополнительно доступны фильтры по типу контента: все, фото, видео, файлы, ссылки, голосовые.

4.5 Уведомления
4.5.1 Push-уведомления
Push-уведомления отображаются при получении нового сообщения, когда приложение свёрнуто или закрыто. Уведомление включает имя отправителя, текст сообщения (если настройки позволяют), звук уведомления. При клике на уведомление открывается соответствующий чат.

4.5.2 Звуковые оповещения
Звук воспроизводится при новом сообщении. Можно выбрать мелодию из предустановленных или отключить. Настраивается громкость. Разные звуки для личных и групповых чатов.

4.5.3 Индивидуальные настройки
Для каждого чата можно настроить индивидуально: включить/выключить уведомления, выбрать звук, настроить вибрацию, разрешить/запретить превью.

5. Нефункциональные требования
5.1 Производительность
Время загрузки списка чатов не должно превышать 500ms при хорошем соединении. Отправка сообщения должна обрабатываться менее чем за 100ms (без учёта сетевой задержки). Интерфейс должен поддерживать 60 FPS при прокрутке и анимациях. Память потребляется не более 200MB при типичном использовании.

5.2 Безопасность
Все коммуникации между клиентом и сервером осуществляются по HTTPS/WSS. Пароли хешируются с использованием bcrypt с стоимостью не менее 10. JWT токены имеют ограниченный срок действия. Refresh token rotation применяется при каждом обновлении. Модель конфиденциальности как в Telegram: облачные чаты защищены шифрованием при передаче и хранении на стороне сервиса (клиент–сервер); настоящее сквозное шифрование сообщений «только у участников» соответствует секретным чатам в Telegram и протоколу MTProto 2.0 (https://core.telegram.org/mtproto, https://core.telegram.org/api/end-to-end) — в текущей версии Системы не реализовано, см. раздел 6. Не использовать в качестве эталона для копирования Telegram протокол Signal (Double Ratchet) — он не является протоколом облачных чатов Telegram.

5.3 Надёжность
Система должна обеспечивать доступность 99.9% времени (допускается не более 8.76 часов простоя в год). Автоматическое переподключение при разрыве соединения. Локальное кэширование для работы оффлайн (чтение истории).

5.4 Масштабируемость
Архитектура должна поддерживать горизонтальное масштабирование серверов. Stateless серверная часть для балансировки нагрузки. Сессии хранятся в Redis для распределения между инстансами.

6. Планы на будущее (функции вне текущей версии)
6.1 Сквозное шифрование (по модели Telegram)
Реализация режима, сопоставимого с секретными чатами Telegram: обмен ключами на клиентах (MTProto Secret Chats / слой https://core.telegram.org/api/end-to-end), шифрование содержимого до отправки на сервер для доставки, опциональная визуализация ключа для сравнения с собеседником (см. https://telegram.org/faq#q-what-is-this-39encryption-key-39-thing). Не смешивать с протоколом Signal: у Telegram иной стек (MTProto 2.0).

6.2 Звонки
Звонки и видеозвонки; в Telegram они заявлены как сквозное шифрование (см. https://telegram.org/faq#q-can-i-make-calls-via-telegram). Технологически — WebRTC или иной стек с учётом требований к безопасности. Push-уведомления для входящих звонков.

6.3 Боты и API
Платформа для ботов по смыслу Telegram Bot API (https://core.telegram.org/bots, https://core.telegram.org/bots/faq). Интеграция внешних сервисов, inline-режим при необходимости (https://core.telegram.org/bots/inline).

6.4 Стикеры и emoji
Библиотеки стикеров. Пользовательские наборы стикеров. Анимированные emoji.

6.5 Игры и Mini Apps
Платформа для HTML5 игр и мини-приложений внутри мессенджера (в экосистеме Telegram — Mini Apps; условия и понятия см. https://telegram.org/tos/mini-apps при проектировании совместимости).
